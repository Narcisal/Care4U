## 一、記憶系統升級

### 現狀
目前記憶系統分兩層：
- `W001_conversation.json`：對話歷史，跨 session 保留，最多 50 則
- `W001.json` 的 `recent_events`：情緒事件紀錄，最多 50 筆

### 問題
- 沒有重要性區分，所有對話和事件同等對待
- 超過上限後舊記憶直接刪除，重要的事可能消失
- 每次對話只注入最近 3 筆事件，AI 能參考的歷史很有限
- 沒有「彙整摘要」機制，無法從大量對話中提取重點

---

### （1）長短期記憶分層

**概念：** 每筆記憶加入 `importance` 重要分數（0.0 ~ 1.0），分數高的升級為長期記憶永久保留，分數低的定期清除。

**資料結構設計：**
```json
{
  "event": "說了：好想念我女兒，她很久沒來了",
  "sentiment": "negative",
  "emotion_score": -0.8,
  "importance": 0.9,
  "memory_type": "long",
  "topic_tags": ["思念", "家人"],
  "date": "2026-04-22",
  "embedding": null
}
```

**重要分數判斷邏輯：**

| 條件 | importance |
|------|-----------|
| 緊急安全事件 | 1.0 |
| 強烈負面情緒（思念、哭泣） | 0.8-0.9 |
| 一般負面情緒 | 0.5-0.7 |
| 正面情緒（開心、感謝） | 0.4-0.6 |
| 日常閒聊 | 0.1-0.3 |

**實作方式：** 在 `llm_service.py` 的 `analyze_emotion` 回傳結果加入 `importance` 分數，由 LLM 判斷，`iSafe` 寫入記憶時一併儲存。

**實作難度：** 低｜預計時程：1 週

---

### （2）記憶整理與彙整

**概念：** 定期把多段對話整理成一個事件摘要，例如：

```
原始對話（5 筆）：
- 今天說了三次想念女兒
- 提到女兒叫小玲
- 說小玲上個月才來過
- 說很想和小玲一起吃飯
- 說希望小玲常回來

彙整摘要：
「王大明爺爺近期多次提到思念女兒小玲，
 上個月曾來訪，希望能更常見面，情緒偏低落。」
```

**實作方式：** 每次對話結束後，若累積超過 10 筆相關事件，自動呼叫 LLM 進行彙整，結果存入 `memory_summary` 欄位。

**對話前撈記憶流程（搭配此功能）：**
```
每次對話前：
  ① 撈最近 5 筆對話紀錄（短期記憶）
  ② 撈相關歷史摘要（中期記憶）
  ③ iSafe 判斷當前情緒
  ④ Decision 決定語調
  ⑤ 注入 System Prompt → MagicAI 生成回應
```

**實作難度：** 中｜預計時程：2 週

### （3）網路搜尋補充長者背景資料

**概念：** 若長者有公開資料（著作、軍職、職業成就），透過搜尋 API 自動取得並存入長期記憶，讓 AI 能更深入理解長者的人生背景。

**使用情境範例：**
- 長者曾任軍官 → 搜尋「XX 軍官 生平」
- 長者曾出版著作 → 搜尋書名、內容摘要
- 長者曾任職某機構 → 補充職業背景

**技術選型：**

| 工具 | 說明 | 費用 |
|------|------|------|
| Tavily API | 專為 LLM 設計的搜尋 API，回傳結構化結果 | 免費額度夠用 |
| Google Custom Search API | 搜尋結果精準，需設定 CSE | 每日 100 次免費 |

**實作流程：**
```
照護人員建檔時勾選「啟用背景資料搜尋」
→ 輸入關鍵字（如姓名、職業、著作）
→ Tavily 搜尋並回傳摘要
→ LLM 過濾不相關資訊
→ 存入 long_term_memory
→ 注入 System Prompt
```

**隱私考量：** 只搜尋公開資料，搜尋前需照護人員確認授權。

**實作難度：** 中｜預計時程：2 週

---

### （4）向量記憶搜尋（pgvector）

**概念：** 現在每次對話會把所有近期事件全部注入 Prompt，未來改用語意向量搜尋，只撈「最相關」的記憶。

**現在 vs 未來：**
```
現在：
  長者說「我想女兒」→ 注入最近 5 筆事件（可能不相關）

未來：
  長者說「我想女兒」→ 向量搜尋「思念、家人、女兒」→ 撈出最相關的歷史記憶
```

**技術：** PostgreSQL + pgvector + Gemini Embedding API

**實作難度：** 高｜預計時程：4 週

## 二、LLM 角色深化

### 現狀
目前 MagicAI 的 LLM 只做一件事：生成對話回應。

### 改良方向

### （1）歷史記憶 RAG（文章型）

**概念：** 將長者過去的對話彙整成文章格式，存成向量索引，每次對話前以 RAG 方式撈出相關段落注入 Prompt。

**與記憶彙整的差異：**
- 記憶彙整：把事件壓縮成摘要（結構化）
- 文章 RAG：把對話整理成自然語言文章，語意搜尋更自然

**實作難度：** 高｜預計時程：3-4 週

## 三、多模態互動

> 對應教授提出的互動增強方向，讓陪伴不只是對話，而是有視覺、聲音的完整體驗。

### （a）懷舊圖片生成

**觸發情境：** 長者說「我好懷念年輕時住的大稻埕舊家」

**流程：**
```
MagicAI 偵測到懷舊話題
→ Decision 呼叫圖片生成工具
→ Gemini Imagen API 根據描述生成圖片
→ 顯示在聊天介面，搭配語音回應
```

**技術：** Gemini Imagen 3 API（已在 Gemini 平台可用）

**實作難度：** 低｜預計時程：1 週

---

### （b）健康衛教影片

**觸發情境：** 長者說「復健動作怎麼做？」

**流程：**
```
iSafe 偵測到健康詢問
→ Decision 判斷類型（復健／用藥／飲食）
→ 搜尋對應的衛教影片連結（YouTube / 衛福部資源）
→ 顯示在聊天介面
```

**說明：** 影片生成目前品質不穩定，先以「搜尋推薦影片」實作，之後再升級為即時生成。

**實作難度：** 低（搜尋推薦）/ 高（即時生成）｜預計時程：1 週（搜尋版）

---

### （c）虛擬家人問候

**觸發情境：** 長者思念女兒小玲

**流程：**
```
MagicAI 偵測到思念家人情緒
→ Decision 呼叫語音克隆工具
→ 使用家人預先錄製的語音樣本（10-30 秒）
→ ElevenLabs 語音克隆生成問候語音
→ 搭配家人照片合成問候畫面
```

**倫理考量：**
- 需要家人明確授權同意
- 生成內容需事先審核
- 建議限制在「問候語」範圍，避免複雜對話

**技術：** ElevenLabs Voice Cloning API + Gemini 圖片合成

**實作難度：** 高｜預計時程：4 週

---

### （d）夢境視覺化

**觸發情境：** 長者說「我昨晚夢到在雲上面種花」

**流程：**
```
MagicAI 偵測到夢境描述
→ LLM 將描述轉化為圖片生成 Prompt
→ Gemini Imagen 生成圖片
→ 顯示在聊天介面
→ MagicAI：「哇，爺爺你的夢好美，在雲上種花，種的是什麼花呀？」
```

**實作難度：** 低｜預計時程：1 週（與懷舊圖片共用架構）

## 四、對話體驗優化

### （1）純語音模式

**概念：** 長者不需要看螢幕、不需要打字，只需要說話和聆聽。

**改動：**
- 自動偵測靜音結束，不需要按按鈕
- AI 回應後自動準備下一次錄音
- 介面簡化為一個大按鈕

**實作難度：** 低｜預計時程：1 週

---

### （2）家人語音克隆

同上方多模態（c），獨立在對話體驗也值得提。長者和 AI 說話時，聽到的是「像家人的聲音」，陪伴感大幅提升。

---

### （3）情緒判斷多元化

**現狀：** 目前用 LLM 分析文字內容判斷情緒。

**改良方向：**

| 方式 | 說明 | 技術 |
|------|------|------|
| 語速偵測 | 說話很快可能緊張，很慢可能疲憊 | 分析音訊時長與字數比 |
| 停頓偵測 | 長時間停頓可能情緒低落 | Whisper 時間戳 |
| 臉部表情 | 配合 YOLO 偵測微笑或皺眉 | OpenCV + 情緒模型 |

**實作難度：** 中（語速）/ 高（臉部）｜預計時程：2-3 週

---

### （4）對話中斷保護

**現狀問題：** TTS 播放期間使用者可以繼續輸入，造成對話錯亂。

**解法：**
```javascript
// AI 回應期間鎖住輸入
async function processAndRespond(message) {
    lockInput()           // 鎖住輸入欄和送出按鈕
    const data = await fetch('/api/chat', ...)
    await speakText(data.message)  // 等 TTS 播完
    unlockInput()         // 解鎖
}
```

**實作難度：** 低｜預計時程：0.5 週

## 五、Agent 架構完整化

### （1）三隻 Agent 協作流程

**現狀：** Decision 協調 MagicAI 和 iSafe，基本流程已完成。

**待完善：**
- iSafe 目前只有文字分析，之後加入臉部情緒偵測
- Decision 的任務排程目前只有介面，實際提醒功能尚未串接
- Agent 間的錯誤處理：若某隻 Agent 失敗，系統應能降級繼續運作

---

### （2）情緒偵測持續優化

**現狀：** LLM 語意判斷，比關鍵字準確。

**改良方向：**
- 加入對話上下文：不只看這句話，同時看前 3 句
- 情緒趨勢分析：連續 3 次負面 → 升高警覺
- 個人化調整：根據長者的說話習慣校準判斷

---

### （3）精油香氛 RAG 整合

**概念：** 根據長者當下情緒，從精油知識庫檢索對應的香氛建議，由 Decision Agent 觸發芳療儀器。

**流程：**
```
iSafe 偵測情緒低落
→ Decision 查詢精油 RAG 知識庫
  （「情緒低落」→ 薰衣草、佛手柑）
→ 回傳香氛建議給照護人員
→ 未來串接智慧香氛儀器自動啟動
```

**知識庫來源：** 已在其他課程完成初步 RAG 實作，可直接整合。

**實作難度：** 低（整合已有 RAG）｜預計時程：1 週
